Introduction

This document is the deep-dive companion to ARCHITECTURE.md. It traces exactly what happens during the life of a Kensington tag instance, from the moment t.div(...) is called until the resulting DOM node and its signal subscriptions are torn down.

You don’t need to read this to use Kensington. Read it if you’re:

  • Hunting a bug in the reactive system
  • Adding a new tag-class variant or rendering mode
  • Designing an integration that needs to understand cleanup semantics
  • Curious how a small library can support signals, SSR, hydration, and reconciliation in ~1200 lines of hand-written source

Throughout this page, code references appear as reactive/signal.js:55. Click to open the source file. Line numbers reflect the source at the time of writing.

Concepts at a glance

If you’ve never read the source before, these are the seven moving parts you’ll see referenced throughout. Each links to the section that explains it in full.

ConceptWhat it isLives in
Tag instance The object returned by t.div(...). Holds attributes, content, namespace, and lifecycle callback arrays. Two output methods: toString() and toElement(). content-tag.js
Signal A reactive value container. .get() subscribes the current effect; .set() schedules every subscriber. .value reads without subscribing. signal.js
effect A closure that re-runs whenever any signal it reads changes. Exposes pause / resume / stop. signal.js:134
Lifecycle Per-element orchestrator that owns every signal effect, the persist mechanism, and the connect/disconnect callback chain. lifecycle.js
DOM tracker A shared MutationObserver that fires stop chains on removal and connect callbacks on insertion. One per document. dom-tracker.js
Reconciler Patches DOM in place when a signal’s value is an array. Matches by data-key, recursive sync, two guards for signal-managed elements. reconcile.js
persist mode Opt-in via toElement({ persist: true }). Effects pause on removal and resume on reconnect. Disconnect/connect callbacks fire every cycle. lifecycle.js
The mental model in three sentences

Every tag is a plain object that becomes a string or an element on demand. When it becomes an element, a Lifecycle wires every signal-driven value into an effect bound to that element via WeakRef. A document-wide MutationObserver watches for that element’s removal and tears the effects down (or pauses them, if persist is on).

Stage 1: Tag Construction

The closure returned by createTag accepts three forms of call:

t.div();                              // no attributes, no content
t.div('hello');                        // content only
t.div({ class: 'a' });                 // attributes only
t.div({ class: 'a' }, 'hello');        // attributes, then content
t.div({ class: 'a' }, [t.p(), t.p()]); // attributes, then array content

The closure body (kensington.js:171) disambiguates these forms by inspecting the first argument’s prototype. If it’s a plain object (Object.prototype or null prototype), it’s treated as attributes; anything else (a tag instance, an array, a string, a number, a Signal) is treated as content.

The createTag closure

Each closure constructs a tag instance with the same template:

return (attributesOrContent = null, content, thirdArg) => {
  // ... argument disambiguation ...
  const instance = new Klass({
    additionalGlobalAttributes: this.additionalGlobalAttributes,
    allowedAttributeMap,    // built once when createTag was called
    attributes,
    content,
    contentIsLiteral,
    encodeContent,
    indentationLevel: this.indentationLevel,
    logger: this.logger,
    namespace,
    namespaces: this.namespaces,
    tagName,
    validationLevel: this.validationLevel,
  });

  if (this.validationLevel !== 'off') {
    instance.validate(this.validationLevel);
  }
  return instance;
};

The allowedAttributeMap is built once when createTag is first called and shared across every invocation of that closure. This is a performance win: validating t.div(…) a million times does not rebuild the spec map a million times.

collectContent

collectContent (content-tag.js:49) recursively flattens nested arrays into a single linear list, dropping items that should not render at all:

function collectContent(items, seen = new Set()) {
  const out = [];
  for (const c of [].concat(items)) {
    if ([undefined, null, '', false, true].includes(c)) {
      continue;  // false/true arise from `someCondition && t.span(...)` patterns
    }
    if (Array.isArray(c)) {
      if (seen.has(c)) { continue; } // circular reuse. Skip
      seen.add(c);
      out.push(...collectContent(c, seen));
      continue;
    }
    out.push(c);
  }
  return out;
}
Key behaviors
  • Boolean false/true are dropped. This is what makes condition && t.span(…) work.
  • null, undefined, and empty string are dropped. Equivalent to “render nothing.”
  • Arrays flatten recursively, and a cycle-detection seen set prevents infinite recursion if a user accidentally creates one.
  • Signals pass through unchanged. They’ll be unwrapped at render time.

Validation

If validationLevel is 'warn' or 'error', instance.validate() runs immediately after construction. The class method delegates to validate.js:96:

  1. Collect unallowed attributes. Filter Object.keys(tag.attributes) through attributeIsValid. An attribute is allowed if it’s on/prop, or it’s in allowedAttributeMap (camelCase or kebab-case), or it matches a namespace prefix (data-, aria-, custom), or it’s in additionalGlobalAttributes.
  2. Report them once via showInvalid. At 'warn' this logs; at 'error' this throws.
  3. Collect invalid attribute values. For each attribute that was allowed, run attributeValueIsValid against the type spec from the map (or the namespace-attr free-pass, or the special-cased id/class/style rules).
  4. Report invalid values as a single combined message. Reporting collects per-attribute strings so the user sees all problems at once, not one per refresh.
Never throws at 'off'

All runtime validation goes through show-invalid.js. At 'off' it’s a no-op. Production deployments run with 'off' for performance; a malformed attribute in user data must not crash the page.

Stage 2: String Output

tag.toString() delegates to serialize.js:47 (renderToString). The path:

  1. Filter invalid content via tag.validateContent(). Items that aren’t a string, finite number, tag instance, or Signal are dropped (and reported via showInvalid).
  2. Open the tag. Concatenate '<', the tag name, the attribute string (with a leading space if non-empty), and '>'.
  3. Render the content body via one of three paths (see below).
  4. Close the tag. Concatenate '</', the tag name, '>'.
Why string concatenation, not template literals?

V8’s rope-string optimization makes many short str += calls faster than a single large template literal interpolation. Kensington’s benchmarks bottlenecked on rendering at one point. See serialize.js:50.

Three content paths

renderToString picks one of three paths based on tag type and content shape:

Path A
Literal content
For <script>/<style> tags (contentIsLiteral). Each child is joined by newlines, and non-script/style get HTML-encoded.
Path B
Short single-line
Fast path when content is a single string/number under 100 characters with no line breaks. Concatenates directly without the stringifyContentArray+indent overhead.
Path C
Multi-line indented
Everything else. Resolves Signals via .get(), flattens, passes to stringifyContentArray, then applies indent at the tag’s indentation level.

The selector is contentIsShort(tag) at serialize.js:9:

export function contentIsShort(tag) {
  if (!tag.content.length) { return true; }
  if (tag.content.length > 1) { return false; }
  let [content] = tag.content;
  if (content instanceof Signal) { content = content.get(); }
  if (!['string', 'number'].includes(typeof content)) { return false; }
  if (content.length > 100) { return false; }
  return !LINE_BREAK_TEST_REGEX.test(content);
}

Attribute serialization

attributeString(tag) at serialize.js:32 calls attributesStringFromObject at attributes.js:82, which iterates the array form and serializes each pair as name="value" (with HTML encoding). Booleans render as the bare name (disabled not disabled="true").

Function values cannot be serialized to strings. When validationLevel !== 'off', the onFunction callback reports them; otherwise they are silently omitted.

Cache check

content-tag.js:137. If #domElement is set, return it. If the cached element is still attached (parentNode !== null), this would silently move the node, which is rarely what the caller wants. showInvalid reports it.

if (this.#domElement) {
  if (this.#domElement.parentNode !== null) {
    showInvalid(`toElement() called on a tag instance already in the DOM …`, ...);
  }
  return this.#domElement;
}
Why cache?

So that getDomElement() returns a stable reference, and so that re-calling toElement() on the same tag instance does not produce two independent DOM nodes that fight over the same signals.

Attribute wiring

The signal-attribute apply function:

lifecycle.signalEffect(attrValue, (el, val) => {
  if (val === false || val === null || val === undefined) {
    el.removeAttribute(attrName);
  } else if (val === true) {
    el.setAttribute(attrName, '');     // bare attribute (disabled, checked)
  } else {
    el.setAttribute(attrName, String(val));
  }
});

The effect runs once immediately to set the initial value, then re-runs every time the signal changes. el is the result of elementRef.deref() inside the lifecycle module. If the element has been garbage-collected, the effect self-stops.

Event handlers (the on object)

content-tag.js:172. The on attribute lets a user attach multiple event handlers via a single nested object:

t.button({ on: { click: handleClick, mouseenter: handleHover } }, 'Press me')

The loop iterates Object.entries(events) and calls element.addEventListener(eventName, handler) for each function value. No cleanup is needed. Event listeners are released when the element is garbage-collected with no other references.

Prop wiring (DOM properties, not attributes)

content-tag.js:181. prop lets a user set DOM properties directly, not attributes. The distinction matters for things like input.value (DOM property reflects current state) versus input[value] (attribute reflects initial state).

t.input({ prop: { value: count } })       // .value updates as `count` changes
t.input({ value: count.get() })            // . Frozen attribute

The dispatch checks isPropWritable against the live element. If the property exists on the prototype chain but is read-only, showInvalid reports it and the assignment is skipped. Otherwise:

if (propValue instanceof Signal) {
  lifecycle.signalEffect(propValue, (el, val) => { el[propName] = val; });
} else {
  element[propName] = propValue;
}

Content wiring

content-tag.js:195. For each item in the (already-flattened) content array, three cases:

Case 1
Tag instance
Recurse into child.toElement() and append the resulting node. The child has its own lifecycle. This parent does not own its cleanup.
Case 2
Signal value
Insert two comment-node anchors. Wire a signal effect that calls reconcile(parent, start, end, items) on every change. Flip hasSignalContent so markContentTracked runs later.
Case 3
Plain value
If a string and not a literal-content tag, apply preserveSpaces. Create a text node, append. String() handles Symbol values gracefully.

The signal-content case is where reactivity meets the DOM. The anchor pair is critical. It gives the reconciler stable boundaries even when sibling content is added or removed around it:

if (node instanceof Signal) {
  hasSignalContent = true;
  const startAnchor = document.createComment('');
  const endAnchor = document.createComment('');
  element.append(startAnchor, endAnchor);
  lifecycle.signalEffect(node, (el, val) => {
    reconcile(el, startAnchor, endAnchor, Array.isArray(val) ? val : [val]);
  });
  continue;
}
Anchors persist for the element’s lifetime

The two comment nodes are held only by the effect’s closure (not by the user). markContentTracked(element) tells the reconciler “don’t replace this element’s children even if a parent reconcile sees a fresh element with different children”. See reconcile.js:51.

Lifecycle finalize

content-tag.js:216. After all wiring, the lifecycle is finalized:

lifecycle.finalize({
  connectCallbacks: this.#connectedCallbacks,
  disconnectCallbacks: this.#disconnectedCallbacks,
  onCleared: () => { if (this.#domElement === element) { this.#domElement = null; } },
  onReconnect: () => { this.#domElement = element; },
});

if (hasSignalContent) {
  markContentTracked(element);
}

this.#domElement = element;
return element;

The four callbacks passed to finalize are what make the lifecycle module reusable without coupling it to ContentTag:

  • connectCallbacks. User-registered via addConnectedCallback. Fire on every insertion (only re-fires on reconnect when persist is true).
  • disconnectCallbacks. User-registered via addDisconnectedCallback. Fire on every removal.
  • onCleared. Internal: reset #domElement to null after removal so a future getDomElement() returns null.
  • onReconnect. Internal: restore #domElement to the live element on re-insertion under persist.

Signal Anatomy

Before tracing the lifecycle module’s internals, here’s how a Signal actually works. The full implementation is ~200 lines at signal.js.

Subscription via .get

A Signal’s subscribers are kept in a private Set on the instance. The mechanism that wires up a subscription is the module-scoped currentEffect reference, set during an effect() or computed() run.

// signal.js:63
get() {
  if (currentEffect !== null && !this.#subscribers.has(currentEffect)) {
    this.#subscribers.add(currentEffect);
    const sub = currentEffect;
    sub._cleanups.push(() => this.#subscribers.delete(sub));
  }
  return this.#value;
}

Key consequences:

  • Calling .get() outside an effect/computed registers no subscription. It’s just a read.
  • Calling .get() twice in the same effect is idempotent. The has(currentEffect) check prevents duplicates.
  • The cleanup function is pushed to the effect’s _cleanups array, which the track helper drains and resets on each re-run.
.value and .toJSON() never subscribe

If you need the current value without registering a dependency (for example, inside an effect body that should not re-run when this particular signal changes), use .value (getter, no tracking) or pass the signal to JSON.stringify (which calls .toJSON(), also no tracking). See signal.js:72 and :104.

Writes and the microtask flush

.set(next) at signal.js:76 compares via Object.is and bails on equality, then iterates this.#subscribers to schedule effect re-runs and immediately call computed-update closures.

sequenceDiagram participant U as User code participant S as Signal participant Q as pending Set participant Mt as queueMicrotask participant E as effect.run U->>S: .set(next) S->>S: Object.is(next, current)? alt equal S-->>U: return early else changed S->>S: private value updated loop each subscriber alt subscriber is effect S->>Q: scheduleRun(fn) S->>Mt: queueMicrotask(flush) else subscriber is computed.update S->>E: fn(value) synchronously end end S-->>U: return Mt->>Q: flush() loop each pending fn Q->>E: run() end end

Effects are batched. Multiple .set() calls in the same synchronous turn coalesce into a single re-run per effect because pending is a Set and scheduleRun is a no-op if the effect is already pending.

Computed updates run synchronously. This is intentional. Computed signals should never “lag” their inputs. A computed reading a.get() + b.get() must always be consistent with the latest values of a and b.

Error isolation in batches

flush() wraps each effect run in try/catch and re-throws via queueMicrotask. This prevents one effect’s thrown error from aborting the batch. Every queued effect still runs. See signal.js:32.

effect

effect(fn) at signal.js:134 creates a tracked closure that re-runs whenever any signal read inside it changes:

export function effect(fn) {
  if (ssrDepth > 0) {
    return { pause() {}, resume() {}, stop() {} };
  }
  let paused = false;
  let destroyed = false;
  function run() {
    if (paused) { return; }
    track(run, fn);
  }
  run._cleanups = [];
  run._isEffect = true;
  run();
  return {
    pause() {
      paused = true;
      pending.delete(run);
      for (const cleanup of run._cleanups) { cleanup(); }
      run._cleanups = [];
    },
    resume() {
      if (destroyed) { return; }
      paused = false;
      run();
    },
    stop() {
      this.pause();
      destroyed = true;
    },
  };
}

The three returned methods give the caller control:

pause

Drains _cleanups (unsubscribing from every signal) and removes itself from pending. The effect won’t re-run until resume() is called. Can be paused/resumed any number of times.

resume

Calls run() immediately, which re-tracks subscriptions to every signal it reads. No-op if destroyed is true.

stop() calls pause() and sets destroyed = true, making resume() a permanent no-op. This is the cleanup path when an element is removed without persist.

The track() helper at signal.js:113 is what sets currentEffect for the duration of a run, ensuring any .get() calls inside subscribe back. It also clears stale subscriptions from the previous run. If an effect reads signalA on one run and not on the next, the subscription to signalA is severed automatically.

computed

computed(fn) at signal.js:184 creates a Signal whose value is derived from other signals. Updates are synchronous (unlike effects).

export function computed(fn) {
  if (ssrDepth > 0) {
    const s = new Signal(fn());
    derivedSignals.add(s);
    return s;
  }
  const s = new Signal(undefined);
  function update() {
    track(update, () => {
      derivedWriteDepth++;
      try {
        s.set(fn());
      } catch (err) {
        rethrowAsync(err);
      } finally {
        derivedWriteDepth--;
      }
    });
  }
  update._cleanups = [];
  update();
  stopFns.set(s, () => {
    for (const cleanup of update._cleanups) { cleanup(); }
    update._cleanups = [];
  });
  derivedSignals.add(s);
  if (currentEffect !== null) {
    currentEffect._cleanups.push(() => s.stop());
  }
  return s;
}

Five small but important details:

  1. SSR bypass. Inside renderForHydration, fn() runs once with no currentEffect set, so source .get() calls do not register a subscription. The returned Signal carries the snapshot value and never updates. This prevents per-request computed/transform calls from leaking subscribers onto module-level signals that outlive the request.
  2. derivedWriteDepth is incremented before the fn() call so that s.set() can succeed even though s is in the derivedSignals WeakSet (which normally blocks external writes).
  3. Errors are caught and re-thrown asynchronously via rethrowAsync so that a throw in one computed doesn’t corrupt other parts of the dependency graph.
  4. stopFns.set(s, …) stores a cleanup closure keyed by the Signal. Signal.stop() at signal.js:95 retrieves and runs it, properly tearing down the derived computation.
  5. Parent auto-stop. If computed is called while another effect’s run or computed’s update is on the stack, this computed’s stop() is pushed onto the parent’s _cleanups. When the parent re-runs (or is stopped), this computed is torn down. Without this, calling computed() in an effect() body grows the source signal’s subscriber set by one per parent re-run.

Signal.prototype.transform = fn => computed(() => fn(this.get())) at signal.js:213 is just sugar for the same pattern. It’s defined on the prototype after computed is in scope to avoid a forward reference.

The Lifecycle Module

esm / lib / reactive / lifecycle.js

createLifecycle({ element, persist }) at lifecycle.js:10 is a closure factory. Each tag’s toElement creates one. The returned object exposes two methods: signalEffect(sig, apply) and finalize({…}).

This module is the entire home of the persist mechanism. No other file decides whether to pause-or-stop on removal.

Internal state

export function createLifecycle({ element, persist }) {
  const stops = [];                              // pause-or-stop closures, one per signal effect
  const resumables = persist ? [] : null;        // effect objects for resume() on reconnect
  const elementRef = new WeakRef(element);       // shared across every signalEffect

  function pauseOrStop(eff) {
    return () => persist ? eff.pause() : eff.stop();
  }

  function wireEffect(eff) {
    stops.push(pauseOrStop(eff));
    if (resumables !== null) { resumables.push(eff); }
  }
  // ...
}

resumables is allocated only when persist is true. The common (non-persist) case has zero overhead.

signalEffect

signalEffect(sig, apply) {
  const eff = effect(() => {
    const el = elementRef.deref();
    if (!el) { eff.stop(); return; }
    apply(el, sig.get());
  });
  wireEffect(eff);
  return eff;
}

The effect runs once synchronously when created, applying the initial signal value. On subsequent runs (triggered by signal writes), the effect dereferences the WeakRef. If the element has been garbage-collected, the effect self-stops. No zombie subscriptions.

WeakRef is the GC safety net

If a user creates a tag, calls toElement, and then loses every reference without ever inserting the element, the element is eligible for GC. The Signal’s subscriber set would otherwise hold the effect closure, which would hold the element by reference. WeakRef breaks the cycle.

finalize

finalize registers the stop chain and (if needed) the connect callback with dom-tracker. Two branches:

persist: false (default)

On removal, every effect’s stop() is called. Permanent teardown. Disconnect callbacks fire once. Connect callback (if any) fires once on the first insertion only.

persist: true

On removal, every effect’s pause() is called. The stop chain is rebuilt for the next cycle via reFireAndRegister. On reconnect, every effect’s resume() is called and the connect callback re-fires.

The disconnect chain

function registerDisconnectChain() {
  trackForStop(element, () => { for (const stop of stops) { stop(); } });
  if (onCleared) { addOnStop(element, onCleared); }
  for (const fn of disconnectCallbacks) {
    addOnStop(element, () => fn.call(element, element));
  }
}

trackForStop registers the first link of the chain (running every signal effect’s pauseOrStop closure). addOnStop appends to that chain. First the onCleared hook (which resets the tag’s #domElement cache), then each user-registered disconnect callback.

The persist rebuild

When persist is true, the chain is rebuilt every cycle so disconnect callbacks fire on every removal, not just the first:

if (persist) {
  const reFireAndRegister = () => {
    trackForStop(element, () => {});                 // fresh empty stop
    if (onCleared) { addOnStop(element, onCleared); }
    for (const fn of disconnectCallbacks) {
      addOnStop(element, () => fn.call(element, element));
    }
    addOnStop(element, reFireAndRegister);             // chain self-perpetuates
  };
  addOnStop(element, reFireAndRegister);
}

The connect path

const needsConnect = persist || connectCallbacks.length > 0;
if (needsConnect) {
  let firstConnection = true;
  trackForConnect(element, () => {
    if (!firstConnection) {
      if (onReconnect) { onReconnect(); }
      if (resumables !== null && resumables.length > 0) {
        for (const eff of resumables) {
          eff.resume();
          addOnStop(element, () => eff.pause());
        }
      }
    }
    firstConnection = false;
    for (const fn of connectCallbacks) { fn.call(element, element); }
  }, persist);
}

The shared callback-fire loop runs on both first connection and reconnection. Only the reconnect-specific work (onReconnect, resume, re-pause wiring) is gated on !firstConnection.

The DOM Tracker

esm / lib / reactive / dom-tracker.js

A shared MutationObserver watches document.documentElement for any subtree mutation. When tracked elements are added or removed, it fires registered callbacks. This is what closes the loop between “element removed from the DOM” and “effects stop, signals unsubscribe.”

The unified entries map

// dom-tracker.js:3
const entries = new Map();
const contentTracked = new WeakSet();
let observer = null;

Each tracked element has a single record in entries:

FieldTypeSet byCleared on
stop () => void trackForStop; extended by addOnStop Element removal (always)
connect () => void trackForConnect Element removal, only if persist is false
persist boolean trackForConnect Element removal, only if persist was false

The entry survives stop-cleanup as long as persist is true and connect is still set. This is how reconnection works. The entry is deleted entirely only when both stop and connect are gone.

The observer

// dom-tracker.js:53
function buildObserver() {
  if (observer !== null) { return; }
  observer = new MutationObserver(records => {
    if (entries.size === 0) { return; }
    for (const record of records) {
      for (const node of record.removedNodes) { stopRemoved(node); }
      for (const node of record.addedNodes)   { fireConnected(node); }
    }
  });
  observer.observe(document.documentElement, { childList: true, subtree: true });
}

The observer is built lazily on first trackForStop/trackForConnect call. There is exactly one for the whole document. Not one per tracked element.

The entries.size === 0 short-circuit means that if every tracked element has been cleaned up, mutation records are skipped without any per-record work. The observer itself remains active because reattaching it would lose the document.body subtree.

The visit helper

Both stopRemoved and fireConnected need to handle two cases for a mutation record’s node: the node itself might be tracked, or it might be an ancestor of one or more tracked elements. visit abstracts this:

// dom-tracker.js:27
function visit(node, fn) {
  if (entries.has(node)) {
    fn(node, entries.get(node));
    return;
  }
  if (node.nodeType !== 1) { return; }              // text nodes etc. have no descendants
  for (const [el, entry] of [...entries]) {         // snapshot copy in case fn mutates
    if (node.contains(el)) { fn(el, entry); }
  }
}
Why the snapshot copy?

stopRemoved deletes from entries inside the callback. Iterating a live Map while deleting from it is allowed in spec but can be subtle. The snapshot makes the iteration order and completeness obvious.

Removal Flow

An element is removed when something calls element.remove(), parent.removeChild(element), parent.replaceChildren(...), etc. The browser fires a mutation record. The shared MutationObserver picks it up.

Step by step

  1. Browser fires the MutationRecord. record.removedNodes contains the directly-removed node(s). The element itself may or may not be the directly-removed node. It could be a descendant of one.
  2. stopRemoved(node) at dom-tracker.js:38. Calls visit(node, fn) which finds the tracked entry for the node itself or any tracked descendant.
  3. clearStop(entry, el). Deletes entry.stop. If not persisted, also deletes connect and persist. If both halves are gone, removes the entry from the map entirely.
  4. The captured stop function runs. This is the chained closure built up via trackForStop + every addOnStop.
  5. For each signal effect: eff.pause() or eff.stop(). Driven by the persist flag. pause drains _cleanups (unsubscribing from each Signal); stop does the same and sets destroyed.
  6. onCleared runs. If the tag’s #domElement still points at this element, set it to null.
  7. Each user disconnect callback runs. In registration order, each fn.call(element, element).
Removal vs stopTracked

The removal path above is triggered automatically by the MutationObserver. stopTracked(el) at dom-tracker.js:100 does the same teardown synchronously without waiting for a mutation record. The reconciler calls this on discarded “fresh” nodes. Nodes that were built up by a parent toElement but then thrown away in favor of patching an existing node.

Persist Mode

By default, removal is permanent: effects stop, the tag’s #domElement cache clears, and the user’s disconnect callbacks fire. If the element is re-inserted later, the effects are gone. The signal subscriptions need to be rebuilt by calling toElement() again.

toElement({ persist: true }) changes this. Effects pause instead of stop. The tag’s #domElement stays cleared during the gap, then restores when the element returns. The disconnect-callback chain rebuilds so it fires on every cycle, not just the first.

This is the pattern you want for elements that move between containers without losing identity. Tabs that swap, modals that hide and reshow, custom elements whose connectedCallback fires multiple times.

sequenceDiagram participant U as User code participant E as Element participant Mo as MutationObserver participant L as Lifecycle participant Sg as Signals Note over E,L: Initial render with persist true U->>E: parent.append(element) Mo->>L: fireConnected, first connect cb run Note over E: User updates signals, effects re-run normally U->>Sg: signal.set(v) Sg->>L: scheduled effect runs Note over E,L: First removal U->>E: parent.removeChild(element) Mo->>L: stopRemoved, each eff pauses L->>L: onCleared (clear domElement) L->>L: disconnectCallbacks fire L->>L: reFireAndRegister installs new stop chain Note over Sg: Subscriptions cleared, .set() during gap does nothing Note over E,L: Re-insertion U->>E: parent.append(element) Mo->>L: fireConnected L->>L: onReconnect (restore domElement) L->>L: for each eff, call eff.resume() (re-subscribes, applies current value) L->>L: addOnStop(eff.pause). Re-arm for next removal L->>L: connect callbacks fire again

The persist invariants

  1. Pause, don’t stop. Every signal effect is captured in resumables. On removal, pauseOrStop picks pause(). The effect closure still exists. It just isn’t subscribed to anything.
  2. Disconnect callbacks re-arm. reFireAndRegister installs a fresh stop chain after each removal so that the next removal fires them again.
  3. Reconnect resumes. eff.resume() calls run(), which re-tracks subscriptions and applies the current signal value. Any updates that happened during the gap are visible immediately.
  4. Resume wires its own pause. Right after eff.resume(), the lifecycle adds () => eff.pause() to the new stop chain. The cycle continues.
  5. Connect callbacks fire every cycle. On the first insertion AND on every reconnect. This is the only way they can fire more than once.

When NOT to use persist

If connectedCallback creates a fresh element each time (the common Web Components pattern), persist is wrong. The old paused effects become orphaned when the new element replaces them. Use persist: true only when you hold the same DOM node across reconnections.

Reconciliation

esm / lib / reactive / reconcile.js

Every signal-content update calls reconcile at reconcile.js:72. The function patches the DOM in place rather than tearing it down and rebuilding. It works for both single-value and array-valued signals. toElement wraps non-arrays as [val] before passing them in, so the algorithm only has to handle the array case.

Reconciliation runs between a pair of comment anchors set up at element construction. The anchors give the function stable boundaries: startAnchor.nextSibling is the first child to consider, endAnchor is the sentinel.

Keyed matching

Each item can have a data-key attribute (or dataKey shorthand). Items with keys are matched against existing children with the same key. Not against the same positional index. This is what enables efficient reordering:

// reconcile.js:72
export function reconcile(parent, startAnchor, endAnchor, newItems) {
  const oldNodes = new Map();
  let node = startAnchor.nextSibling;
  while (node !== endAnchor) {
    const key = node.dataset?.key;
    if (key !== undefined) { oldNodes.set(key, node); }
    node = node.nextSibling;
  }
  // ...
}

Without a key, items are matched positionally (and recreated if shape differs).

Snapshot fast path

Once a keyed match is found, the reconciler checks a structural snapshot of the previous render before doing any DOM work. A WeakMap keyed by DOM node holds the last (attributes, content) pair that produced it. If the new tag’s attributes and content are value-equal to the snapshot, the entire itemToNode(item) + syncNode chain is skipped. The existing DOM node is reused as-is.

// reconcile.js
if (snapshotMatches(snapshots.get(old), item)) {
  // Attributes and content structurally equal the previous render. Skip the
  // toElement() build and the syncNode diff. Just reuse the existing node.
  targetNode = old;
} else {
  targetNode = syncNode(old, itemToNode(item));
  recordSnapshot(targetNode, item);
}

valueEqual handles plain objects and arrays by structure, recurses into ContentTag children (matching on tagName + attributes + content), and falls back to reference equality for everything else (functions, Signal, LiteralTag, CommentTag, DOM nodes, Date, anything with a non-plain prototype).

Why value equality, not reference equality

The natural pattern arr.map(item => t.li({ class: item.cls }, item.label)) allocates a fresh attribute object literal on every render. Reference equality on those literals would always miss. Value equality detects that the literal is structurally identical to last time and skips the rebuild without forcing the developer to memoize.

A stable Signal reference (typed in once and stored on the item) hits the fast path because it satisfies the reference-equality fallback. A fresh closure or fresh LiteralTag per render does not. The snapshot is recorded only on the non-fast-path branch, so an item that hits the fast path keeps its original snapshot indefinitely.

There is a known circular import here. reconcile.js imports ContentTag for the instanceof check, and content-tag.js imports reconcile for its signal-content effect. Rollup emits a CIRCULAR_DEPENDENCY warning that is informational only. Both sides consult the other inside function bodies that run at call time, not at module-load time, so ESM live bindings resolve correctly.

syncNode. The recursive patcher

syncNode(existing, fresh) at reconcile.js:19 handles a matched pair. If the node types differ, the fresh node replaces the existing one. If they’re both text nodes, just nodeValue is patched. If they’re both elements:

  1. Apply every fresh attribute. Remove any old attribute not present in fresh.
  2. Recursively syncNode each child pair.

The two guards

Why guards are needed

The “fresh” node passed to syncNode is the result of calling itemToNode(item), which calls item.toElement(). That fresh node is fully wired with its own signal effects pointing at the fresh element. If we patched attributes/children naively, we’d corrupt the live element’s effects.

Two guards prevent this:

// reconcile.js:43
if (!isTracked(existing)) {
  for (const attr of oldAttrNames) {
    existing.removeAttribute(attr);
  }
}
// reconcile.js:51
if (!isContentTracked(existing)) {
  const oldChildren = [...existing.childNodes];
  const newChildren = [...fresh.childNodes];
  // ... positional sync ...
}
  • isTracked(existing). If true, the existing element has signal-managed attributes. Don’t remove any attribute that wasn’t on the fresh node. Those signal effects haven’t yet applied their initial values to fresh when reconcile inspects it.
  • isContentTracked(existing). If true, the existing element holds signal-content comment anchors. Don’t patch its children at all. Replacing the anchors would break the live content effects whose closures still reference them.

After patching, stopTracked(fresh) tears down the discarded fresh node’s effects so they don’t leak.

Insertion, reuse, leftover cleanup

Back in reconcile() itself, the main loop walks newItems in order. For each item:

  1. Skip null/undefined/false items.
  2. If the item has a key and matches an existing keyed node, call syncNode(old, itemToNode(item)). The result is either the patched existing or the fresh node if types diverged.
  3. If no match (or no key), build a brand-new node via itemToNode(item).
  4. If cursor === targetNode, advance. The node is already in position. Otherwise parent.insertBefore(targetNode, cursor) to slide it into place.
  5. After the loop, every node between cursor and endAnchor is leftover from the previous render. Remove them.
  6. Every entry remaining in the oldNodes map is a keyed node whose key was not in newItems. Remove them too.

SSR and Hydration

esm / lib / render / hydration.js

On the server (or any environment without a real DOM), reactive subscriptions must not be created. They would have nothing to update and no cleanup path, so they would leak immediately. Both effect() and computed() consult the ssrDepth counter at signal.js:5.

The SSR bypass

export function effect(fn) {
  if (ssrDepth > 0) {
    return { pause() {}, resume() {}, stop() {} };
  }
  // ... normal path ...
}

export function computed(fn) {
  if (ssrDepth > 0) {
    const s = new Signal(fn());     // value-only snapshot
    derivedSignals.add(s);
    return s;
  }
  // ... normal path ...
}

Inside an SSR call, effect() returns a no-op stub and computed() returns a frozen-value Signal. No subscriptions are created in either case. tag.toString() still reads signal values via .get() (which works fine without a current effect) and produces a static HTML snapshot.

Why computed() needs the bypass too

Without it, a per-request computed (or transform) reading a module-level signal would register an update closure in that signal’s subscriber set. The signal lives across requests, so each render leaves one dead subscriber behind. Over time the subscriber set, and the time spent iterating it on every set(), grows without bound.

renderForHydration

renderForHydration(html, components, state) in hydration.js wraps an SSR render. It increments ssrDepth, invokes each component function in the state map, embeds the resulting HTML alongside a JSON state script block, and decrements ssrDepth in finally.

registerComponents + the JSON block

On the client, registerComponents({ name: fn }) reads the JSON block, looks up each registered component by name, and runs fn(state) to produce a fresh tag instance. The new tag’s toElement() creates the live DOM tree (with signal effects this time), which then replaces the SSR-rendered HTML node in the document.

This is “remove and replace” hydration, not “reuse and attach.” The SSR HTML is for time-to-first-paint; the live version takes over once JS is ready.

Why not reuse?

Reuse hydration is feasible but requires the SSR HTML and the client’s tag-tree shape to match exactly. The current strategy avoids that constraint at the cost of one extra DOM swap per component.

Invariants

The rules that hold across every code path. Violations are bugs.

  1. validationLevel: 'off' never throws on runtime input. All validation routes through show-invalid.js, which is a no-op at 'off'. Only hard invariants (e.g. createTag called with a non-string tagName) throw unconditionally.
  2. Signal values are accepted everywhere a plain value is accepted. attributeValueIsValid returns true for Signals without inspecting them. Resolution happens at render time, not validation time. See validate.js:56.
  3. .value and .toJSON() do not subscribe; .get() and .toString() do. The asymmetry is intentional. Lets you read a current value inside an effect without creating a dependency. Tests cover this behavior.
  4. The persist mechanism lives entirely in lifecycle.js. No other file decides between pause() and stop(). The dom-tracker knows about persist only to decide whether to preserve the connect/persist entry fields across stop-cleanup.
  5. The two reconcile guards must never be bypassed. isTracked prevents attribute-strip on signal-managed elements. isContentTracked prevents child-patch on elements holding signal-content anchors. See reconcile.js:43.
  6. Effects batch via microtasks; computed updates are synchronous. Multiple .set() calls in the same turn coalesce into one effect re-run. Computed signals see consistent inputs because their updates happen inline with the write.
  7. The DOM tracker has exactly one observer for the whole document. Not one per tracked element. It’s built lazily on the first trackForStop or trackForConnect call.
  8. WeakRef is the GC safety net for signal effects on detached elements. If an element is never inserted into the DOM and is garbage-collected, the next signal write triggers an effect that finds ref.deref() returning undefined and self-stops.
